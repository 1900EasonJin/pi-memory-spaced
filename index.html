🔷 Pi Agent 记忆扩展生态

Pi Coding Agent 的设计哲学是"内核最小，一切皆扩展"。记忆也不例外——官方不内置记忆模块，但社区提供了 至少 13 个 记忆相关扩展，覆盖从纯文件到向量搜索的各个层次。

📦 Pi Memory (jayzeng) ★ 最活跃推荐

npm 包 pi-memory，基于 qmd（轻量搜索引擎）的语义搜索记忆系统。四个核心工具 + 可选搜索，Markdown 文件存储。

📝 核心工具
memory_write — 写入长期记忆或每日日志
memory_read — 读取记忆文件
scratchpad — 清单管理
memory_search — 搜索（需 qmd）
memory_status — 健康检查
⚡ KV 缓存稳定快照
默认使用缓存稳定模式，在 session_start、session_before_compact、长期写入时刷新快照。避免每轮重建导致 KV 缓存失效。
🔍 搜索三模式
keyword (~30ms) BM25 精确匹配
semantic (~2s) 向量语义搜索
deep (~10s) 混合搜索+重排序

📦 @pukljak/pi-memory

持续学习的记忆系统。四个能力闭环：学习→存储→调整质量→注入上下文。

  • 命令体系：/memory.status / /memory.search / /memory.timeline / /memory.ui（本地 Web UI）
  • 高级模式：Playbook（规则/偏好管理）+ Superpowers 工作流支持
  • 质量门控：冲突检测、低质量自动抑制、结果反馈学习（接受/拒绝）
  • 私密标记：private-tagged 内容在持久化前脱敏

📦 tickernelz/pi-memory

三角色文件注入系统：MEMORY.md（长期记忆）+ IDENTITY.md（AI 人格）+ USER.md（用户画像）+ daily/（每日日志）。

  • 首次运行交互式引导（Bootstrap 流程）
  • 自动将 MEMORY.md / IDENTITY.md / USER.md 注入系统提示
  • Dashboard 部件可选启用
🔗 claude-mem 系（3 个变体）
claude-mem 桥接 FTS5+Chroma
HaqimIskandar/pi-agent-memory、ArtemisAI/pi-mem、husniadil/pi-mem — 将 claude-mem（55k+ stars）移植到 Pi Agent。
FTS5 全文搜索 + Chroma 向量嵌入混合检索，跨引擎共享记忆。
🧠 三层架构系（2 个）
分层记忆 图谱增强
Hdaisen/pi-memory-system：Core Prompt → Session Notebook → Turn Summary
becksclair/pi-memory：Session 层 → Durable 层 → Graph 层（实体-声明图）
🌲 层次树系
无限容量 零依赖
ArtiNexus/pi-memory-tree：WATCH.md（焦点板）→ MEMORY.md（关键词索引）→ Level 1+N（详细分支）。文件系统原生层次树，自动级联召回。
🗄️ 持久智能 + 治理
Obsidian 集成 审查回溯
Mont3ll/pi-persistent-intelligence：可治理持久记忆，会话搜索、审查式策展、可选的 Obsidian Vault 集成。
⚡ Pi Agent Memory (chendpoc)
秘密脱敏 本地优先
本地 Markdown 记忆，自动脱敏常见密钥，跨会话偏好/约定/决策记忆。
🔄 自动学习系
LLM 提取 会话末尾整理
davehardy20/pi-memory、bscott/pi-memory、samfoy/pi-memory：会话结束时通过 LLM 自动提取偏好、模式、纠正，注入下个会话。

💡 Pi Agent 记忆生态特点： 完全去中心化、社区驱动。大多数方案基于纯 Markdown 文件存储（可读、可编辑、可 Git 管理），部分可选接入向量搜索。没有"标准"方案，适合按需选用。

🟣 Claude Code 记忆系统

Claude Code 拥有目前 AI 编码 Agent 中最完善的记忆体系之一。两个互补的记忆系统协同工作。

系统一：CLAUDE.md 文件（手动编写）

📄 四级作用域
Managed Policy — 团队统一管理策略
User — ~/.claude/CLAUDE.md 用户全局指令
Project — ./CLAUDE.md 或 ./.claude/CLAUDE.md
Local — ./CLAUDE.local.md 本地覆盖
📦 存储位置
位于 ~/.claude/projects/<name>/memory/
MEMORY.md 作为索引文件，最多 200 行 / 25KB
每个记忆是独立的 Markdown 文件，含 YAML frontmatter

系统二：自动记忆（Auto Memory）

Claude 在会话中自主判断哪些信息值得记住，自动写入记忆文件。模型自主决定何时写入、写什么。

  • 存储位置：~/.claude/projects/<repo>/memory/
  • 索引：MEMORY.md 始终加载到上下文（首 200 行 / 25KB）
  • 检索：通过标准文件工具读取，而非向量检索
  • 4 个记忆类型：user_role / feedback / reference / ...
  • 作用域：每个仓库独立

💡 Claude Code 记忆精髓： 手动编写（CLAUDE.md）+ 自动积累（Auto Memory）双轨并行。索引文件（MEMORY.md）始终加载，详细记忆按需读取。不依赖向量数据库，纯文件系统方案。

🟢 Codex CLI 记忆管线（OpenAI）

Codex CLI 的记忆系统是目前技术最成熟的方案之一。基于 Rust 实现的多阶段管线，具备差异遗忘等高级特性。

┌─────────────────────────────────────────────────────┐
│              Codex CLI 记忆管线                       │
├─────────────────────────────────────────────────────┤
│  Phase 1: Startup Extraction                          │
│  ┌─────────────────────────────────────┐             │
│  │ 扫描最近会话记录（Rollouts）          │             │
│  │ 提取 → 去重 → 秘密脱敏 → 持久化      │             │
│  └─────────────────────────────────────┘             │
│           │                                          │
│           ▼                                          │
│  Phase 2: Global Consolidation                       │
│  ┌─────────────────────────────────────┐             │
│  │ 跨会话全局整合 → 冲突解决              │             │
│  │ → 差异化遗忘（Diff-based Forgetting）  │             │
│  └─────────────────────────────────────┘             │
│           │                                          │
│           ▼                                          │
│  Context Injection                                    │
│  ┌─────────────────────────────────────┐             │
│  │ 使用率感知的检索 → 注入当前上下文      │             │
│  │ 私密信息自动脱敏                      │             │
│  └─────────────────────────────────────┘             │
└─────────────────────────────────────────────────────┘
  • Phase 1：启动时从近期会话记录（Rollouts）中提取记忆，去重、秘密脱敏后持久化
  • Phase 2：全局整合与冲突解决，差异遗忘算法自动淘汰过时事实
  • 上下文注入：使用率感知的智能检索，私密信息自动脱敏
  • 存储：~/.codex/memories/ 全局统一存储（v2 迁移至此）
  • 命令：/m_update 手动更新记忆

💡 Codex CLI 记忆特色： 多阶段管线架构（提取→整合→注入）+ 差异化遗忘算法。秘密脱敏层是唯一在产品级做了敏感信息自动过滤的方案。

🟤 Cursor 记忆方案

Cursor 本身未内置记忆系统，依赖 MCP（Model Context Protocol）生态提供第三方程。主流方案如下：

📦 cursor-mem
自动记录编辑、Shell 命令、MCP 调用 → 注入 Cursor Rules。零配置，无需 API Key。
📦 0Latency Memory
MCP 服务端，免费 1000 条记忆。通过 remember/recall 工具访问。
📦 cursor-mem0
基于 mem0 管线，本地 embedding（fastembed）+ Qdrant 向量存储，可选 MCP 接口。
📦 Walrus Memory
通过 MCP 提供 memwal_remember / memwal_recall 等工具，语义搜索 + 生命周期钩子。

此外，Agent-Memo.AI 等跨平台方案也支持 Cursor。

🌊 Windsurf Cascade 记忆

Windsurf 提供了双机制记忆系统：

📝 Cascade Memories
Cascade 自动生成的会话记忆。每个对话回合中自动更新，跨会话持久。
可通过 @memory 命令访问和管理。
📄 Rules（.windsurfrules）
用户手动编写的规则文件，永久有效。类似 .cursorrules，存入仓库。
每次会话启动时读取，整个团队共享。

💡 Windsurf 特色： 自动记忆（Memories）+ 手动规则（Rules）双机制。Memories 是 AI 生成的会话上下文，Rules 是人类编写的持久指令。但记忆局限在编辑器内，不能跨工具共享。

🔶 Cline — Memory Bank 方法论

Cline 本身没有内置记忆模块，但社区发展出著名的 Memory Bank 方法论——通过结构化 Markdown 文件让 Agent "永远不忘"。

┌──────────────────────────────────────────────────┐
│              Memory Bank 架构                      │
├──────────────────────────────────────────────────┤
│  projectbrief.md  — 项目概述与核心目标             │
│  productContext.md — 为什么做、解决什么问题           │
│  systemPatterns.md — 架构模式与技术选型             │
│  activeContext.md  — 当前焦点与最近变更             │
│  progress.md      — 进度追踪与待办项               │
│  decisionLog.md   — 关键决策与理由                │
└──────────────────────────────────────────────────┘
  • 实现方式：通过自定义指令（Custom Instructions）+ 生命周期钩子
  • 触发：会话启动时自动加载，监视文件变更
  • 可选增强：Hindsight 集成（自动回忆 + 保留）、MCP Memory Bank 服务器

💡 Cline Memory Bank 价值： 方法论而非工具。将"记忆"变成一种可复制的文档规范，适合所有文件型 Agent 采纳。

🤖 Aider 记忆方案

Aider 通过 Hindsight 包装器实现跨会话持久记忆：

  • hindsight-aider 是 aider 命令的透明包装
  • 会话前：从记忆库召回相关上下文，注入只读文件
  • 会话后：保留完整会话记录供下次提取
  • 每 Git 仓库独立作用域
  • 同样有 Stompy 等替代方案

💡 Aider 方案特色： 无侵入包装器模式——不修改 Aider 本身，通过环境变量和文件实现记忆。

🧩 通用 Agent 记忆框架

以下为独立于特定 Agent 的通用记忆系统，可通过 MCP/SDK 集成到各 Agent：

📦 Mem0
即插即用5行代码
最快的持久记忆方案。自动提取用户偏好，语义搜索 + 遗忘管理。支持 Pi Agent 插件、MCP 接入。
📦 Zep
企业级图谱记忆
实体关系图记忆 + 会话历史 + 用户画像。GraphRAG 增强，企业级部署。
📦 Letta（原 MemGPT）
有状态 Agent自编辑
将 Agent 本身变为有状态服务，记忆可自我编辑和更新。虚拟内存管理，自主运行场景首选。
📦 Cognee
知识图谱
基于知识图谱的记忆引擎。适合文档型记忆需求：实体、关系、结构化检索。
📦 LangMem (LangChain)
LC 生态
LangChain 生态的记忆方案，与 LangGraph/LangSmith 深度集成。
📦 Hindsight
会话前后包装
会话前召回 + 会话后保留，无侵入包装。支持 Cline、Aider 等，通过生命周期钩子实现。

💡 2026 年基准发现： Zhou 等人的论文 "Are We Ready For An Agent-Native Memory System?" 评估了 12 个记忆系统，发现记忆架构的选择对成本和性能影响巨大。Mem0 在快速集成上领先，Zep/Letta 在复杂场景表现更好。

🏗️ 记忆架构模式总结

纵观所有方案，AI Agent 记忆系统可归纳为 4 种架构模式：

📁 文件系统模式
最简单
代表：Claude Code MEMORY.md、Pi Memory、Cline Memory Bank
纯 Markdown/文本文件，Agent 通过标准文件工具读写。
✅ 透明可审计、Git 友好、零依赖
❌ 检索能力有限、无语义理解
🔍 混合检索模式
最流行
代表：pi-memory+qmd、claude-mem、Hindsight
文件存储 + BM25/向量混合检索。
✅ 兼顾透明度和检索能力
❌ 需要额外服务/依赖
🏭 管线处理模式
最复杂
代表：Codex CLI Memories、Mem0
多阶段管线：提取→清洗→整合→注入→遗忘。
✅ 自动化程度最高、质量门控完善
❌ 实现复杂、运维成本高
🏛️ 有状态服务模式
最先进
代表：Letta、Zep
Agent 本身是有状态服务，记忆是第一公民。
✅ 记忆一致性最好、可扩展性强
❌ 架构开销大、不适合轻量场景 
📁 文件系统           🔍 混合检索          🏭 管线处理         🏛️ 有状态服务
                      简单的               中等                复杂               极高

🔮 趋势与建议

2026 年记忆系统趋势

  • 从附属到第一公民：记忆不再是"附加功能"，而是 Agent 架构的核心组件。Letta 的有状态 Agent 理念正在被广泛采纳。
  • 遗忘与遗忘同等重要：Codex CLI 的差异遗忘、Mem0 的自动淘汰——好的记忆系统必须知道该忘记什么。
  • MCP 成为标准接口：Cursor、Windsurf、Cline 等通过 MCP 接入通用记忆服务，跨平台复用。
  • 混合搜索是标配：纯文件检索正在被 BM25+向量混合搜索取代。
  • 秘密脱敏不可忽视：Codex CLI 的秘密脱敏层是一个被低估但至关重要的设计。

如果你要为 Pi Agent 开发记忆扩展……

建议参考的架构：

1️⃣ 底层用 Markdown 文件存储（透明可审计）
2️⃣ 检索层用 qmd/qdrant 实现 BM25+向量混合搜索
3️⃣ 参考 @pukljak/pi-memory 的质量门控机制（冲突检测、反馈学习）
4️⃣ 学习 Codex CLI 的管线思路（提取→整合→注入→遗忘）
5️⃣ 提供 Web UI 方便用户审查和编辑（参考 @pukljak 的 /memory.ui）
6️⃣ 通过 Pi Extension 的事件系统（session_start / session_before_compact / tool_call hooks）实现非侵入集成
